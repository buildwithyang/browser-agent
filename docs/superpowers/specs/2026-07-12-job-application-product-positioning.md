# 求职申请工作流产品定位

日期：2026-07-12

## 决策

Agent Bridge 的近期产品定位从“通用网页上下文投递工具”收敛为“基于 LinkedIn、Indeed 等岗位页面生成定制化求职申请的 Browser Agent”。

第一条完整工作流为：

```text
岗位页面 → JD 匹配分析 → 定制化 CV → Cover Letter → 投递记录 → 模拟面试 → Offer
```

## 愿景

大愿景是尊重用户的注意力，让 AI 成为工作流的一部分。浏览器仍是用户获取信息的主要入口，Agent 应理解用户明确关注的页面，并直接在当前工作流中提供帮助。

小愿景是先解决高频且结果明确的求职需求：帮助用户读懂 JD、判断是否匹配，并准备申请材料，最终帮助用户获得 Offer。

## 当前范围

- 从招聘网站页面显式采集 JD 上下文。
- 使用当前登录用户的生效 CV 进行岗位匹配。
- 输出岗位与业务介绍、真实匹配评分、技能命中与差距。
- 用户确认岗位值得申请后，按需生成 Cover Letter 和 CV 修改建议。
- 保持云端、多租户和用户数据隔离设计。

## 路线图边界

当前“CV 能力”是修改建议，不应在用户文档中描述为已经能够生成完整定制化 CV 文件。

后续依次补齐：

1. 基于用户真实经历生成定制化 CV。
2. 保存岗位、投递状态及对应版本的 CV / Cover Letter。
3. 基于 JD 与 CV 进行模拟面试和反馈。

通用文章、GitHub Issue、邮件等网页场景仍属于长期愿景，但不再占据近期 README 的主叙事。

## 文档约束

- `README.md` 与 `README.zh-CN.md` 必须保持相同的产品定位和能力边界。
- 面向用户优先使用 LinkedIn / Indeed、JD、CV、Cover Letter、Offer 等求职语言。
- 清楚区分“当前能力”和“路线图”，避免把未实现能力写成现状。
